## 二、核心设计原则

### 2.1 夹逼管道（Code → LLM → Code）

和嘟嘟 L2 工作流一样，SOP Swarm 也遵循"确定性夹逼非确定性"：

```
阶段 0 (Searcher)    → 确定性: 搜到什么就是什么, 不添加
阶段 1 (Generator)   → 非确定性: LLM 生成初稿
阶段 2 (Red Teamer)  → 确定性: 逐条交叉验证 facts_anchor
阶段 3 (Architect)   → 非确定性: LLM 重构终稿
                     但被阶段2的审计意见严格约束
```

**关键**: 阶段 2 是 Swarm 的"免疫系统"——它不产生内容，只执行否决。任何被阶段 2 标记为"无事实支撑"的内容，阶段 3 必须删除。

### 2.2 事实边界（Facts Anchor）

为什么单独的 AI 写研报经常胡编？因为它用预训练权重"推演"数据。

SOP Swarm 的解法: **在 LLM 开始写作之前，先用搜索锚定事实边界**。阶段 0 的 `facts_anchor.md` 就是一面墙——Generator 只能在这面墙里面写。墙外的东西，它不知道也不能编。

### 2.3 反 VC 幻觉协议（Anti-BS Protocol）

逐字扒掉宇树第一版的问题:

| 幻觉类型 | 示例 | 审计结果 |
|---------|------|---------|
| 无根基比喻 | "宇树是具身智能时代的 Android" | ❌ 宇树没有 OS/平台/生态，不配比 Android |
| 虚构能力 | "UnifoLM 自研具身智能基础模型" | ❌ 无 benchmark/demo/论文，PR 叙事不能当事实 |
| 强行拔高 | "开发者生态锁入" | ❌ 只有基础 SDK，不是平台，不是生态 |
| 省略限定词 | "5,500 台出货 = 规模效应" | ❌ 绝对值 $100M 在制造业中不算规模 |

修正后的原则: 没有就是没有。护城河可以写"暂无"。

## 三、输出标准

### 7 模块强制模板

| # | 模块 | 要求 |
|---|------|------|
| 1 | 一句话定位 | 极度克制，只描述它在解决什么问题 |
| 2 | 用户画像 | 表格: 用户群 / 痛点 / 使用场景 |
| 3 | 核心功能拆解 | 表格: 功能 / 做对了什么 / 真实短板 / 严重程度 |
| 4 | 商业模式 | 含 Mermaid 流程图 + LaTeX 公式 |
| 5 | 竞争格局 | 表格对比 ≥3 个竞品（开源/闭源分列） |
| 6 | 护城河判断 | 分层分析 + 倒推验证 + 可写"暂无" |
| 7 | 一句话判断 | 基于事实的独立判断，不含比喻 |

### 信源标注规范

全文所有数据点强制标注:
- `[已核实]` = 多源交叉验证
- `[官方宣称]` = 公司自述但未独立验证
- `[推测]` = 基于公开数据的合理估算
- `[缺乏公开数据]` = 无可靠信源
