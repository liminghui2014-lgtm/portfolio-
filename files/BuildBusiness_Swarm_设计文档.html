BuildBusiness Swarm · 四 Agent 产品构建流水线

灵感 → 商业计划 → PRD → 代码 → 审计。四名独立 Agent 协同，禁止跳步。

Agent 架构
当用户想设计任何产品时主动触发。强制商业验证在前，代码开发在后。审计不通过最多打回修复 3 轮。

设计动机

独立开发者的最大成本不是写代码——是写了一个没人要的东西。

BuildBusiness Swarm 的核心假设：在投入一行代码之前，先验证这个产品能不能卖钱。四名 Agent 各司其职、不可跳过——商业模式没想清楚之前，PM 不接手；PRD 没被确认之前，Hacker 不动键盘；最后一道关卡是商业级审计，代码必须达到"能拿出去卖钱"的标准才算通过。

核心设计原则

1. 商业先于代码（Absolute Priority）

Biz_Strategist 没有点头之前，一行代码都不准写。这是整个 Swarm 的宪法。Product Hunt 上 90% 的失败产品，问题都不在代码质量——在于一开始就没想清楚谁付钱。

2. 无情裁剪（Ruthless Scope Slashing）

Indie_PM 的唯一KPI：一周内能交付。任何对"促成用户首次付费"没有直接帮助的功能——砍掉。PRD 超过两页就是过度设计。

3. 审计独立（Independent Audit）

Auditor 只能报问题，不能修代码。Hacker 必须亲手修复每一个 Error_Log 条目。审计三要素：致命缺陷（白屏/崩溃）、商业级体验（Loading/Hover/人话报错）、安全性（越权/注入/明文泄露）。

4. 打回循环（Feedback Loop）

Hacker → Auditor → 不通过 → Hacker → Auditor → ... 最多 3 轮。3 轮后仍不通过，Router 通知人类介入决策。避免无限循环消耗 Token。

级别含义示例
🔴 致命阻塞上线：白屏、崩溃、数据丢失API 返回 null 未处理导致页面白屏

触发规则

当用户表达"想做一个东西""设计产品""搭个XX工具"时，主控 Router 自动询问是否启动 Swarm。手动触发："BuildBusiness 构建 {产品名}"。
